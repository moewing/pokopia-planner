## 🎯 游戏机制关键洞察（规划器算法基础）

### 资源循环在单一环境里都不能自给

这是数据统计出来的硬事实，**但在正确的"每格子独立环境"机制下不是问题**：

| 循环 | 环境自给情况 | 解决方案 |
|---|---|---|
| 木材循环 | 仅昏暗可自给（靠双斧战龙 #73 唯一乱撒小圆木）| 昏暗地图主跑此循环 |
| 红砖循环 | 任何单一环境都无法自给 | 一张地图放多种环境格子（温暖格子住点火，潮湿格子住乱撒黏土） |
| 铁条循环 | 跨 3-4 环境 | 同上，一张地图内混搭多种环境 |

这个洞察**验证了"每格子独立环境"机制的设计**——否则资源循环根本没法跑。

## 💡 给建站的关键建议

### 1. 数据过滤原则

```typescript
// 规划器/相似度/资源循环只用可入住的
const forPlanning = pokemon.filter(p => p.is_playable);  // 287 只

// 图鉴展示所有
const forPokedex = pokemon;  // 300 只(不可入住的加标记)
```

### 2. 规划器约束优先级

```
约束 1（硬）: 每张地图 ≤25 只
约束 2（硬）: 同一循环的宝可梦在同一张地图
约束 3（软）: 同一格子内的宝可梦喜欢同一环境（舒适度加分）
约束 4（软）: 重叠区物品覆盖尽量多只
约束 5（软）: 不同循环分到不同地图（避免点火互抢）
```

### 3. UI 交互重点

- 规划器主视图应该是 **5 个地图 Tab**，不是单一画布
- 每张地图 Tab 显示：人数 X/25，承载的循环，格子布局
- 图鉴里不可入住的 13 只应该有视觉区分（比如淡灰色 + 🚫图标）
